# ADR 002: State Management for Human-in-the-Loop and Audit Logging

**Date:** 2026-03-27
**Status:** Accepted

## Context
The system must temporarily suspend destructive infrastructure commands (e.g., stopping a container) while awaiting human approval. Additionally, every proposed action, approval, and rejection must be logged immutably. Because multiple LLM agents might propose actions simultaneously, our state management and logging mechanisms must be strictly thread-safe and non-blocking.

## Decision
1. **Audit Log (Concurrency):** We implemented the Actor Model using Kotlin Coroutines (`Channel<PlanOfAction>`). A single background coroutine reads from this channel and writes to the log sequentially.
2. **Approval Gate (State):** We implemented an asynchronous approval registry using `ConcurrentHashMap<String, PlanOfAction>`. 

## Rationale
* **Actor Model over Mutex Locks:** Instead of using heavy thread locks (`synchronized` blocks) to prevent race conditions when writing to the audit log, the Actor model uses a queue (Mailbox). Any thread can fire a message into the channel instantly, and the single Actor processes them one by one. This ensures sequential logging with zero blocking.
* **In-Memory Thread-Safe State:** `ConcurrentHashMap` provides native thread-safety for adding and removing pending actions.
* **Fail-Safe Design:** Holding pending state in-memory means that if the controller crashes, pending actions are lost. In an infrastructure context, this is the desired fail-safe behavior: an action must be explicitly re-proposed by the agent rather than executing a stale command after a reboot.

## Consequences
* **Positive:** High throughput for audit logging without thread contention.
* **Positive:** Asynchronous design prevents the MCP server from timing out while waiting for a human to click "Approve".
* **Negative:** Pending requests do not survive server restarts (accepted trade-off for safety).
